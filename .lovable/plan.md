
# Agent Reports: Action Button Redesign

## Overview
Replace the current Agent Reports action buttons (Mark Reviewed, Validate, Dismiss) with a new workflow-oriented button set that ties reports to actionable outcomes.

## Current vs New Button Logic

| Current | New | Purpose |
|---------|-----|---------|
| Mark Reviewed | **Escalate as Outage** | Auto-generates an Outage Request for attendance-related violations |
| Validate | **Validate (Action for Coaching)** | Confirms the report as valid and flags for coaching follow-up |
| Dismiss | **Dismiss (Invalid Report)** | Marks the report as an invalid/false positive |

## Escalation Rules

**Incident Type Mapping:**
- `LATE_LOGIN` → Creates outage with reason `"Late Login"`
- `TIME_NOT_MET` → Creates outage with reason `"Undertime"`
- `EARLY_OUT` → Creates outage with reason `"Undertime"`

**Request Details:**
- Status: `for_review` (same as system auto-generated requests)
- Uses incident date as the outage date
- Calculates duration from report details (schedule vs actual times)
- Sets `is_auto_generated: false` (admin-initiated, not system-triggered)
- Links to the agent's client/team lead/role from their profile

**Report Status After Escalation:**
- Status changes to `reviewed` (renamed to "Escalated" in UI labels)

## Technical Implementation

### Phase 1: API Layer Updates

**File: `src/lib/agentReportsApi.ts`**
1. Update `STATUS_CONFIG` labels:
   - `reviewed`: Change label from "Reviewed" to "Escalated"
   - `validated`: Change label to "Validated"
   - `dismissed`: Change label to "Dismissed"

2. Add new helper function `isEscalatableIncident(type: IncidentType)`:
   - Returns `true` for `LATE_LOGIN`, `EARLY_OUT`, `TIME_NOT_MET`
   - Returns `false` for other incident types

3. Add new function `escalateReportToOutage(report: AgentReport, escalatedBy: string)`:
   - Maps incident type to outage reason
   - Extracts time data from `report.details`
   - Fetches agent profile for client/team lead/role
   - Creates outage request via existing `createAutoGeneratedLateRequest` (with minor modifications)
   - Updates report status to `reviewed`
   - Returns success/error result

**File: `src/lib/leaveRequestApi.ts`**
1. Add new function `createEscalatedOutageRequest(input: EscalatedOutageInput)`:
   - Similar to `createAutoGeneratedLateRequest` but with configurable `outage_reason`
   - Sets `is_auto_generated: false` to distinguish from system-generated
   - Returns the created request

   - For non-escalatable types, show a disabled state or hide entirely
   - Icon: `ArrowUpRight` or `ExternalLink` to indicate outage creation

2. Update button text and icons:

### Phase 3: Button Visibility Logic

**Escalate as Outage Button:**
- Visible only for: `LATE_LOGIN`, `EARLY_OUT`, `TIME_NOT_MET`
- Enabled when report status is `open` or `reviewed`
- Hidden for other incident types (Quota, Gap, Bio, Restarts, Overbreak, No Logout)

**Validate Button:**
- Always visible for open/reviewed reports
- Represents coaching action needed

**Dismiss Button:**
- Always visible for open/reviewed reports
- Represents false positive / invalid report

### Data Flow for Escalation

```text
1. Admin clicks "Escalate as Outage" on a LATE_LOGIN report
2. System extracts from report.details:
   - scheduleStart: "09:00"
   - loginTime: "09:25"
   - incidentDate: "2026-02-05"
3. Fetch agent_profiles for client/team_lead/role
4. Create outage request:
   - agent_email: report.agent_email
   - start_date: "2026-02-05"
   - end_date: "2026-02-05"
   - start_time: "09:05" (schedule + 5min grace)
   - end_time: "09:24" (1 min before actual login)
   - outage_reason: "Late Login"
   - status: "for_review"
5. Update report status to "reviewed"
6. Show success: "Outage request LR-0123 created"
```

## Edge Cases Handled

1. **Missing profile data**: If agent profile not found, fall back to using report data and show warning
2. **Duplicate outage check**: Verify no existing outage for same agent/date/reason before creating
3. **Time calculation edge cases**: Handle overnight shifts and missing time data gracefully
4. **Non-escalatable incidents**: Hide escalate button for types like QUOTA_NOT_MET, HIGH_GAP, etc.

## Summary Card Updates

The "Reviewed" summary card will be renamed to "Escalated" to reflect the new workflow. The count will still track reports with `status = 'reviewed'`, but the UI label will make it clear these are reports that triggered an outage request.
